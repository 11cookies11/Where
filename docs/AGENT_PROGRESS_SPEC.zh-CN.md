# Where Agent 写入规范（仅 Markdown）

本规范用于指导本地 LLM agent 写入 `where.sourceFile`（默认 `.where-agent-progress.md`），让 Where 插件稳定解析并展示进度。

## 1. 源文件位置

- 文件路径来自 VS Code 配置：`where.sourceFile`
- 默认路径：`.where-agent-progress.md`（工作区根目录）
- 编码：`UTF-8`

## 2. 唯一支持格式（Markdown）

```md
# Plan: <计划标题>
- [ ] <任务A>
  - [x] <子任务A1>
  - [~] <子任务A2>
- [~] <任务B>
- [!] <任务C（阻塞原因可写在标题）>
- [x] <任务D>
```

硬性规则：

- 第一行建议为标题：`# Plan: <title>`
- 任务行必须匹配：`- [<状态>] <标题>`
- 子任务使用缩进表达层级（建议每级 2 个空格）
- 每行一个任务，不要写多行任务
- 不要输出 JSON
- 不要在文件中写无关长段落（会降低解析稳定性）

状态枚举：

- `[ ]` => `todo`
- `[~]` => `in_progress`
- `[!]` => `blocked`
- `[x]` => `done`

## 2.1 多层级与“伸缩（折叠/展开）”写法

Where 的树形“折叠/展开”由任务缩进自动推断，不需要额外语法。

- 正确做法：用空格表达父子关系，建议每级固定增加 2 个空格
- 禁止做法：用 Tab 缩进（会触发 warning）
- 不推荐：奇数空格缩进（1/3/5 空格，层级容易不稳定）
- 不推荐：跨级跳缩进（例如父级 0 空格，子级直接 6 空格）
- 不需要 `<details>`、`<summary>` 或其他自定义折叠标记

推荐示例（Codex 最容易稳定遵守）：

```md
# Plan: API Refactor
- [~] 重构 API 层
  - [x] 拆分路由定义
  - [~] 提取鉴权中间件
    - [ ] 覆盖异常分支
  - [ ] 更新集成测试
- [ ] 同步文档
```

反例（容易被误解析或告警）：

```md
# Plan: Bad Example
- [~] 父任务
<TAB>- [ ] 使用 Tab 缩进（错误示意）
   - [ ] 3 个空格缩进（不推荐）
      - [ ] 6 个空格跳级（不推荐）
<details><summary>自定义折叠</summary></details>  <!-- 不会被识别为任务 -->
```

## 3. Agent 更新策略（建议）

- 已有任务状态变化时，优先更新原任务，避免重复追加同名任务
- 新任务追加到末尾
- 标题简短可执行，避免模糊措辞
- 阻塞任务在标题内写原因，例如：`等待 API Key 审批`

## 4. 给本地 LLM agent 的提示词模板

可将下面内容直接作为系统/开发指令：

```text
你负责维护项目进度源文件（where.sourceFile）。
输出必须严格遵守 Where 插件可解析 Markdown 格式。

要求：
1) 第一行：# Plan: <title>
2) 后续每行：- [ ] / - [~] / - [!] / - [x] + 空格 + 任务标题
3) 只允许四种状态：
   [ ] todo, [~] in_progress, [!] blocked, [x] done
4) 保持 UTF-8 文本；不要输出 JSON；不要输出无关解释性段落。
5) 状态变化时修改原任务，避免重复任务。
6) 阻塞项标题内写明阻塞原因。
7) 多层级仅使用空格缩进，建议每级 2 空格；不要使用 Tab 或 `<details>` 折叠语法。
```

## 5. 多 Agent 适配

- 本规范是“通用协议”，不绑定具体模型厂商。
- 可在仓库 `prompts/` 目录为不同 Agent 维护轻量提示词模板（如 Codex、Claude、Gemini、Qwen）。
- 各模板必须遵循本规范；若模板和本规范冲突，以本规范为准。
